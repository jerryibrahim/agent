# AI Coding Agent Guidelines

## Operating Principles

- **Correctness over cleverness.** Prefer boring, readable solutions that are easy to maintain.
- **Smallest change that works.** Minimize blast radius; don't refactor adjacent code unless it meaningfully reduces risk or complexity.
- **Leverage existing patterns.** Follow established project conventions before introducing new abstractions or dependencies.
- **Prove it works.** "Seems right" is not done. Validate with tests/build/lint.
- **Be explicit about uncertainty.** If you cannot verify something, say so and propose the safest next step.

---

## Git Commits

- Do not append Co-Authored-By lines to commit messages.
- Never commit the `.vscode/` directory.
- Commit messages must be multiline:
  - First line: the short description provided by the user.
  - Blank line, then a bullet-point body auto-generated by you summarizing the key changes.
- Keep commits atomic and describable; avoid "misc fixes" bundles.
- Don't mix formatting-only changes with behavioral changes.

## Workflow

### 1. Plan First

Enter plan mode for any non-trivial task (3+ steps, multi-file change, architectural decision).

1. **Read first.** Read all files that will be affected. Do not propose changes to code you haven't read.
2. **Write a plan** to `tasks/todo.md` describing:
   - Context, objective, and acceptance criteria (what must be true when done).
   - Files to create (with a one-line description of each).
   - Files to modify (with a one-line description of each change).
   - Key design decisions as a table (Decision | Choice | Rationale).
   - Verification steps (build, test, manual checks).
3. **Get approval.** Present the plan before implementing. If new information invalidates the plan, stop, update it, then continue.
4. **Write a spec first** when requirements are ambiguous (inputs/outputs, edge cases, success criteria).

### 2. Break Into Discrete Tasks

Each task should be independently implementable, testable, and committable. Map dependencies so independent tasks can run in parallel across sessions.

```text
  Database migration (independent)
       ↓
  [Provider impl, Worker binary] (parallel, independent of each other)
       ↓
  Integration tests (depends on both above)
```

### 3. Incremental Delivery

- Prefer thin vertical slices over big-bang changes.
- Implement → test → verify → then expand.
- When scope reveals deeper issues, fix only what is necessary for correctness; log follow-ups as TODOs rather than expanding the current task.

### 4. Subagent Strategy

Use subagents to keep the main context clean and to parallelize:

- Repo exploration, pattern discovery, test failure triage, dependency research.
- Give each subagent **one focused objective** and a concrete deliverable.
- Merge subagent outputs into a short, actionable synthesis before coding.

### 5. Verification Before "Done"

- Never mark complete without evidence: tests, lint, build, or a deterministic manual repro.
- Ask: "Would a staff engineer approve this diff and the verification story?"

## Task Management

Use the TodoWrite tool to track progress on every non-trivial change:

1. **Plan** — Write a checklist to `tasks/todo.md` with explicit verify tasks. Include a timestamp (`YYYY-MM-DD HH:MM`) at the top of each new plan.
2. **Acceptance criteria** — Define what must be true when done.
3. **Track** — Mark items `in_progress` when starting, `completed` immediately when done. Keep exactly one `in_progress` at a time.
4. **Checkpoint notes** — Capture discoveries, decisions, and constraints in `tasks/todo.md` as you learn them.
5. **Results** — After completing changes, add a "Results" section: what changed, where, how verified. Include a completion timestamp.
6. **Changelog** — Update `tasks/todo_changelog.md` with files created/modified and one-line descriptions, grouped by logical feature. Each feature group must include a timestamp (`YYYY-MM-DD HH:MM`) indicating when the changes were completed.
7. **Lessons** — After corrections or mistakes, update `CLAUDE_lessons.md` with: failure mode, detection signal, prevention rule. Review it at session start and before major refactors.

## Communication

- **Concise, high-signal.** Lead with outcome and impact, not process. Reference concrete artifacts (file paths, commands, error messages).
- **Ask only when blocked.** Ask exactly one targeted question, provide a recommended default, and state what changes based on the answer.
- **State assumptions.** If you inferred requirements, list them briefly.
- **Show verification.** Always include what you ran (tests/lint/build) and the outcome. If you didn't run something, give a command the user can run.
- **Skip busywork updates.** Don't narrate every step. Provide checkpoints when scope changes, risks appear, verification fails, or a decision is needed.

## Error Handling and Recovery

### Stop-the-Line Rule

If anything unexpected happens (test failure, build error, behavior regression): stop adding features, preserve evidence, return to diagnosis and re-plan.

### Triage Checklist

1. **Reproduce** reliably.
2. **Localize** the failure (which layer: handler, service, provider, store, build tooling).
3. **Reduce** to a minimal failing case.
4. **Fix** root cause, not symptoms.
5. **Guard** with regression coverage.
6. **Verify** end-to-end for the original report.

### Autonomous Bug Fixing

- Reproduce → isolate root cause → fix → add regression coverage → verify.
- Do not offload debugging to the user unless truly blocked.

## Engineering Practices

### Code Style

- Go 1.22+ stdlib router patterns (method-based routing with `{name}` wildcards).
- Go 1.26+ will be the default version of go
- Structured logging via `slog`. No external logging libraries.
- Configuration via environment variables. No config libraries.
- Error handling: wrap errors with `fmt.Errorf("context: %w", err)`.
- Tests are collocated with source (`*_test.go` in the same package).
- Prefer explicit names and direct control flow. Avoid clever meta-programming unless the project already uses it.

### Markdown Style

- Linter config: `.markdownlint.json` at the repo root (MD013 line-length and MD060 table-style are disabled).
- Fenced code blocks must always have a language tag. Use `text` for diagrams, flow charts, directory trees, and plain-text output. Use the correct language (`go`, `sql`, `json`, `bash`, `yaml`) for code.
- Bare URLs must be wrapped in angle brackets: `<https://example.com>`.
- Lists and fenced code blocks inside list items must be surrounded by blank lines.

### Testing Strategy

- Add the smallest test that would have caught the bug.
- Prefer unit tests for pure logic, integration tests for DB/network boundaries.
- Avoid brittle tests tied to incidental implementation details.
- Avoid noop test stubs

### Dependency Discipline

- Do not add new dependencies unless the existing stack cannot solve it cleanly.
- Prefer standard library and existing utilities.

### Type Safety

- Encode invariants where they belong: validation at boundaries, not scattered checks.

## Architecture

```text
HTTP Request → Middleware → Handler → Service → Router → Provider → Store
```

- **Handlers:** HTTP concerns only (status codes, serialization).
- **Services:** Business logic, validation, state machines.
- **Router:** Binding resolution (template → implementation → provider).
- **Provider:** Engine-specific translation (Temporal SDK, Trigger.dev API).
- **Store:** Type-safe queries via sqlc.

## Commands

### Build and Test (always safe to run)

- `go build ./...` — compile everything
- `go test ./... -v -race` — run all tests
- `go vet ./...` — static analysis
- `make build` — build the API binary
- `make test` — run tests
- `make lint` — run go vet

### Development

- `make dev` — start DB, migrate, build, run
- `make db-start` — start PostgreSQL
- `make migrate-up` / `make migrate-down` — run migrations

### Docker

- `make docker-build` — build images
- `make docker-down` — stop all services

### Code Generation

- `make sqlc-generate` — regenerate Go from SQL queries

## Definition of Done

A task is done when:

- Behavior matches acceptance criteria.
- Tests/lint/build pass (or there is a documented reason they were not run).
- Code follows existing conventions and is readable.
- A short verification story exists: what changed + how we know it works.
- `tasks/todo_changelog.md` is updated.
